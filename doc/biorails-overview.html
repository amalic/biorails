Overview

Biorails is aimed as helping in the management of projects with data capture of structured data ,linked to a stong set of content management features to enable the anonation of captured data and presentation of texual and image data.


As a systems is really splits down into a number of separate parts.

  1. Catalogue

  2. Project Folders

  3. Inventory of materials

  4. Orgnization of data

  5. Execution of tasks


In version 1.0 the focus was simple web based data capture, in the current version 2.0 management of projects and unstructured data was added to round out the product. Now is is based seen a project folder based data capture front-end.

Catalogue

This is the system administrator workstop. The key change there is the management of lookups and links to remove systems. The system is built with integration in mind and the catalogue is the key point for definition of read only references to other systems. It is expected that most inventory and some other key lookups will be defined in other systems.

Data Concept

This namespace represents a tree of concepts describing the business data. By default a initial Biorails namespace is provided to work in. This Conceptual namespace is used to define the general framework for catagorization of data. Concepts would be items like :-

  • Colour

  • Compound

Concepts are linked to parameter types defined by the administrator to limit the list of lookup the user can use for a parameter.

Data Element

This is actual implementation of these concepts has been separated out as Data element. These provide simple lookup list implementation of concept. The separation may initially seem like overkill, but is aimed at address the key problem multiple implementation. For the example of same samples, there may be implementation my compounds, external compound, natural products. nitially the following styles of lookup list can be defined as a Data Element

  • lists (simple 1,2,3,4 style lists managed internally)

  • models ( in Coded Ruby models)

  • SQL Views (Named table or views in systems)

  • SQL select (Hand crafted SQL selects)

In use these would be thinks

  • Colours in english

  • Internal registered compounds

Data System

The implementation of any Data Element is based on the rules defined in Data System. This defines how to connect and talk to the system. In concept there is a Initial internal systems (biorails this self) then other remote database schema can be registered for use as read only lookup. Currenlty this supports Oracle,MySql,Postgres,Sqlite databases.

For a sceening operation this would be something like the followig:-

    Parameter Roles

These are used to partition the parameters be used so a subset can be easliet picked up for further use. As present the systems comes with a default list of :-

  • Results

Project

This deals wiith the general partitioning of work and access control. Access to a project is controlled via membership, only members see or know a project exists in the front end. The project is moduled on the idea of a folders to match how many people organization there.


The system automatically created folders for each study design and experiment run as part of the project. People can also create other folder to hold notes, files and links to specific items in biorails.

Folders can contains the following:-


  • Articles, html texual content

  • Asserts, external images, files etc

  • References, these are links to other items in biorails (like tasks, reports etc)

  • sub Folders


As a general rule articles/assets are versioned to changes over time can be tasked. This is a general approach taken throughout the application. Folders can be used in a number of ways:-


  • Directories like file system

  • Component Documents with each element chained together

  • Blogs where items are display in reverse order of time

One of the aims of the folder was to allow easy reuse of blocks of text in report. With this aim the systems allow simple drag and drop of items into a folder and there reordering to create a long document or report.


Items like studies, experiments,task as automatic rendering so they can be simply dropped on into a folder to get a summary of there details in the document. Also the inbuilt simple report builder/runner allows the saving of a report to as a article.




Inventory

At present this areas is filled by simple default implementation for common inventories. In most cases there is likily to be some customization to integrate with other systems here unless you only need simple compounds, batch etc. In fact implimentation of new inventory types is possibilty the best intoduction to rails coding in biorails.


Ogranization

This deals with study designs and can be split into three parts as follows:-


  • Parameter space

  • Services provided

  • Protocol Definitions


Parameters

This is the usage of a parameter type with a predefined Lookup or Format. In the study design the scientist gets to define the list of parameters they use, based on the catalogue. This is a process of selecting and parameter type, chosening the format , default units and lookup to use and finally alias if needed. The allow a parameter type like RawData to be converted into a more defined definition like like rate in unit of cpm with format double 5.2. This add validation rules and units to data.


Services

This is basically a named queue for a type of data element acepted into the study. Examples could be a queue of plates or samples. This is the forefilement side of requesting and allows the external services provided my a study to be defined in terms of stuff that is sent to it. For items on a service queue can then be mantained (accepted,rejected etc) by a member of the project with the correct rights.


Protocols

This is basically a number of linked grouping of parameters (contexts in biorails tems). The context a linked in a tree to generate more complex parent/child trees if needed. This allows the system to easily build complex data entry tree / grid like plate->compound->well.


For people more used to older more limited systems like excel templates this can be simply viewed in terms a context as a row and parameter as a column in sheet and to define them parameters are dragged from a tree to column headering.


This is a extra important link to queues here. As in a queue linked parameter is used in a column then as items are added as the protocol in run in a task then the queue_items are linked and automatically updated to match the task status. This is aimed a getting the software do to the hard work of maintaining the current status of all requested items
